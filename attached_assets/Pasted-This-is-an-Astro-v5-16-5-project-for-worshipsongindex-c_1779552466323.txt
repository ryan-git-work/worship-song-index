# Step 2 — Add Organization + WebSite schema to the base layout

In the base layout (so it renders on every page), inject these two JSON-LD blocks in the <head>:

**Organization schema:**

```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "Worship Song Index",
  "url": "https://worshipsongindex.com",
  "logo": "https://worshipsongindex.com/[whatever-the-actual-logo-path-is].png",
  "description": "A free, structured worship-song reference for church worship leaders. Search 3,470+ worship songs by key, BPM, time signature, scripture, and theme."
}
```

**WebSite schema (with SearchAction if the site has a search input):**

```json
{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "name": "Worship Song Index",
  "url": "https://worshipsongindex.com",
  "potentialAction": {
    "@type": "SearchAction",
    "target": {
      "@type": "EntryPoint",
      "urlTemplate": "https://worshipsongindex.com/search?q={search_term_string}"
    },
    "query-input": "required name=search_term_string"
  }
}
```

If the site does NOT have a search endpoint yet, omit the `potentialAction` block and just include `@type: WebSite` with name and url.

# Step 3 — Add BreadcrumbList schema to every page that has visible breadcrumbs

The recon shows breadcrumbs already render on every page (Home / Browse / [page]). For each route template that displays breadcrumbs, inject a JSON-LD BreadcrumbList block matching the visible breadcrumb hierarchy. Example for a song detail page at /songs/so-be-it-elevation-worship:

```json
      "@type": "ListItem",

```

Apply the same pattern to browse pages, where position 3 is the browse-type label (e.g., "Time Signature: 6/8") and the item URL is the browse page URL.

# Step 4 — Add MusicComposition schema to every song detail page

In `src/pages/songs/[slug].astro` (or equivalent), generate a JSON-LD MusicComposition block for each song using the song record's data. Use this template — only populate fields that exist in the data, omit any that are null:

```json
{
  "@context": "https://schema.org",
  "@type": "MusicComposition",
  "name": "{song.title}",
  "url": "https://worshipsongindex.com/songs/{song.slug}",
  "composer": [
    { "@type": "Person", "name": "{songwriter1}" },
    { "@type": "Person", "name": "{songwriter2}" }
  ],
  "musicalKey": "{song.male_key} (male), {song.female_key} (female)",
  "beatsPerMinute": {song.bpm},
  "inLanguage": "en",
  "recordedAs": {
    "@type": "MusicRecording",
    "name": "{song.title}",
    "byArtist": { "@type": "MusicGroup", "name": "{song.artist}" }
  },
  "about": [
    { "@type": "Thing", "name": "{theme1}" },
    { "@type": "Thing", "name": "{theme2}" }
  ],
  "citation": [
    "{scripture_ref_1}",
    "{scripture_ref_2}"
  ]
}
```

Notes:
- If the data doesn't have separate songwriter records, omit the `composer` array entirely. Don't fabricate.
- `about` should pull from the song's themes array.
- `citation` should pull from the song's scripture_refs array.
- `musicalKey` is unusual to express as "X (male), Y (female)" but the spec allows any string — this is more useful for worship-leader searchers than picking one.
